# 架构与代码逻辑

> BrightStar 培训助手 —— AWS Serverless 微信 agent。本文讲整体架构、数据流、数据模型、代码结构与关键逻辑。

---

## 一、整体架构

```
  学员(微信) ─┐                         ┌─ DynamoDB ×6 (Students/Courses/Enrollments/
              │  微信客服(kf)            │            Groups/Results/Knowledge)
  老师(企业微信)┤                         │
              ▼                         ▼
   [企业微信平台] ──回调(加密事件)──► [API Gateway HTTP API] ──► [Lambda: webhook]
              ▲                                                    │  │
              │  出站调用(sync/send)                               │  └─► Bedrock (Titan 向量 / Claude 生成, jp.境内)
              │                                                    │
        [中转 relay] ◄── Lambda 出站(可信IP) ── (Tailscale Funnel) ┘
     sons02: wechat_relay.py
     https://sons02-relay.tail5a0084.ts.net  出口IP 111.104.33.44(可信IP白名单)

  课程网站(浏览器) ──/web/login,/web/submit...──► [API Gateway] ──► [Lambda: web] ──► DynamoDB(Students/Results)

  [EventBridge rate(10min)] ──► [Lambda: reminder] ──► 经中转(微信)/LINE push 给报名学员推送开课提醒(多平台扇出)

  学员(LINE) ──LINE Messaging API──► [API Gateway /line] ──► [Lambda: line-webhook]
       └─ 署名(HMAC-SHA256)验签のみ・中継/AES不要 → 复用 webhook._route → 同一 DynamoDB / Bedrock
```

要点：
- **入站**：企业微信把加密事件 POST 到 API Gateway → webhook Lambda 验签解密。**LINE 则 POST 到 `/line` → line-webhook 验 HMAC-SHA256 签名**。
- **出站**：微信客服(kf) 的拉取/回复都是出站 API，必须从“可信IP”发出 → 经 **sons02 中转**（Tailscale Funnel 暴露入口，中转出口 IP 在白名单）。Lambda→中转带共享密钥 `X-Relay-Auth`。**LINE 出站直连 `api.line.me`，无需可信IP/中转/AES。**
- **多平台共用**：路由核心 `webhook._route` 平台无关；微信与 LINE 各自只有一层适配（验签/事件/回复），后端业务/RAG/提醒**一套代码**。发送统一经 `messaging.send`，按 openid 前缀 `line:` 自动选 LINE push / 微信客服。
- **账号关联**：同一人跨平台用「绑定 <登录码>」合并（`resolve_openid`）；提醒按主账号 `get_channels` 向各平台扇出。
- **AI**：webhook 调 Bedrock（Titan 向量 + Claude 生成，`jp.` 日本境内，数据不出日本）。
- 全 Serverless、按需计费、无固定服务器。

## 二、消息数据流

### 微信客服(kf) 文本消息
1. 企业微信回调 `POST /wechat`，body 是加密事件 `kf_msg_or_event`（含 Token + OpenKfId）。
2. webhook 验 `msg_signature` → AES 解密 → 识别为 kf 事件。
3. `handle_kf`：经中转 `sync_msg` 拉取消息（按 cursor 翻页）→ 逐条：
   - 时效过滤（>5 分钟跳过）、`claim_msgid` 幂等去重；
   - 文本 → `_route` 生成回复；enter_session → 引导注册；
   - 经中转 `send_text` 回复。
4. 立即回 200（kf 异步，不受 5s 被动回复限制）。

### 自建应用被动消息
`POST /wechat` 解密 → `_route` → `build_encrypted_reply` 加密 XML 同步返回（须 5s 内）。

### URL 验签（GET）
`GET /wechat?...&echostr=密文` → 验签 → 解密 echostr → 回明文。

## 三、数据模型（DynamoDB，全部按需 + SSE）
| 表 | 主键 | GSI | 用途 |
|---|---|---|---|
| Students | `openid` | — | 学员档案：name/status/role/lang/aiMode/loginCode/loginCodeExp。特殊键：`__logincode__<码>`(反向索引)、`__kfcursor__`、`__kfmsg__<id>`(去重) |
| Courses | `courseId` | — | 课程：title/startTime(UTC)/durationMin/capacity/enrolledCount/status/zoom*/remind1hSent |
| Enrollments | `openid`+`courseId` | GSI1: `courseId`→`openid` | 报名记录（status=enrolled/cancelled） |
| Groups | `courseId`+`groupId` | — | 随机分组结果 |
| Results | `openid`+`itemKey` | GSI1: `itemKey`→`openid` | 成绩/问卷，itemKey=`quiz#lesson1`/`survey#lesson1` |
| Knowledge | `docId`+`chunkId` | — | RAG 知识块：title/text/vec(base64 float32) |

> 注册状态机：`awaiting_lang` → `awaiting_name` → `active`。

## 五、关键逻辑

### 路由 `webhook._route`（顺序很重要）
事件(enter/click) → 老师命令(白名单，先于注册) → 注册流程(选语言/填姓名) → 切换语言 → 登录码 → 改名 → 老师认证 → 老师命令鉴权 → **AI 开关** → 数字快捷 → 意图解析。
- **AI 模式**：开启后，识别到的具体操作(报名/列课/数字)照常执行；只有“自由提问”(intent=help) 才走 `rag.answer`。回复统一前缀 `@姓名`。

### 双语 i18n
学员档案存 `lang`（默认日语）。所有面向学员的回复经 `T(lang,key)`；命令支持中日别名（报名/申込、取消/キャンセル、改名/名前変更、AI/AI終了…）。

### 登录码（7 天）
`get_or_create_login_code`：无/过期则换发新码、写反向索引 `__logincode__<码>`(含 exp) 并删旧；`find_by_login_code` 校验未过期。历史无 exp 的码向后兼容(不过期)。

### 跨平台账号关联（LINE/微信 同一人）
不同平台给同一个人的 ID 天生不同，无法自动识别。解法：复用「登录码」作本人凭证手动合并。
- `_route` 开头先拦截「绑定/連携 <码>」(用**原始** openid)，调 `business.link_by_login_code`：`find_by_login_code(码)` 找到主账号 → 把当前次账号写成别名记录 `{openid, linkedTo: 主账号, status:linked}`，并把该 openid 登记到主账号 `channels`。
- 其余消息：`_route` 用 `resolve_openid(原始openid)` 解析——次账号(有 linkedTo)→主账号，之后注册/课程/老师权限/登录码全部按主账号处理。
- 提醒：`reminder` 对每个报名按 `get_channels(主账号)`（主账号 + 各平台 channels）逐一 `messaging.send`，**微信+LINE 同时触达**。
- `teacher.list_students` 排除别名(linkedTo)与保留键(`__`)，名单不重复。

### 发送层（`messaging.py`）
`send(openid,text)`：`line:` 前缀 → `line.push`(直连 api.line.me)；否则 → `kf.send_text`(经中转)。业务/提醒只调它，无需关心平台。

### LINE 适配（`line.py` / `line_webhook.py`）
`verify_signature`(HMAC-SHA256(channel_secret, 原始body) == X-Line-Signature) → `parse_events`(归一成 _route 的 msg：文本/follow) → `webhook._route` → `reply`(reply token，免费不计配额)。无 AES、无中转、无可信IP。

### RAG（`rag.py`）
`embed`(Titan v2, 256 维, 归一化) → `retrieve`(scan Knowledge + 点积余弦取 top-k, 阈值过滤) → `answer`(把命中块拼进 system+user，调 `BEDROCK_CHAT_MODEL_ID` 生成，按 lang)。任一步失败都回友好兜底，不抛异常。向量小量级用暴力余弦即可，无需 OpenSearch。

### 提醒（`reminder.py`）
EventBridge 每 10 分钟触发，扫开课时间落在 [now+55, +65] 分钟且未发过的已发布课程，幂等占位 `remind1hSent` 后给报名学员经中转推送（按学员 lang）。

### 中转（`relay/wechat_relay.py`）
轻量 HTTP：`/kf/*` 转发到 `qyapi`，复用 `requests.Session`(keep-alive) + 保活线程降延迟；设 `RELAY_AUTH` 后校验 `X-Relay-Auth`（公网暴露防滥用）；access_token 内存缓存。凭证由 Lambda 每次请求带上，**中转本机不存密钥**。

## 六、部署机制
- **SAM**：`aws cloudformation package`(打包 src/→S3) + `deploy`(服务端跑 SAM 转换)。机密走 `--parameter-overrides`(NoEcho)。
- **CDK**(`cdk/`)：等价 Python 栈，资源自动命名、`table.grant_*` 自动授权(含 index)、HTTP API CORS 自动处理 OPTIONS；机密走 `-c` context。默认 `stage=cdk` 与线上 SAM `dev` 独立。
- 两者共用同一份 `src/` Lambda 代码与 `tools/`、`tests/`。

## 七、设计取舍（为什么这么做）
- **微信客服 kf + 中转**：要让学员在“微信”里对话且满足可信IP，必须出站经固定 IP 中转。
- **Bedrock + 自建 RAG（不用 OpenSearch）**：数据小、无语义大检索需求；OpenSearch 月费 $350+ 不划算。Bedrock 按 token、`jp.` 境内合规。
- **DynamoDB 按需 + 全资源打标签**：零固定费、便于按 `Project=brightstar` 统一查找/删除。
- **纯 Python AES**：免 pycryptodome/Docker，Lambda 零第三方依赖，打包即压缩 src/。
- **平台无关路由 + 适配层**：`_route` 不依赖具体平台，新增渠道（LINE）只写一层薄适配，后端零改动；LINE 因协议简单（仅签名）几乎零成本接入。同一人跨平台用登录码关联，而非强制账号体系（无需手机号/邮箱）。
