# BrightStar 社内システム ｜ アーキテクチャ

フルサーバーレス（固定サーバなし・従量課金・自動スケール）。IaC は **AWS CDK (TypeScript)**、ランタイムは **Python 3.12 / arm64（外部依存ゼロ）**。リージョンは東京（ap-northeast-1）。

---

## 全体構成
```
                          ┌─────────────── AWS (ap-northeast-1) ───────────────┐
  LINE 公式アカウント      │                                                     │
  （人事アシスタント）      │   Lambda Function URL ──► webhook (Python3.12/arm64) │
        │  webhook(POST) ─────────►  ├─ 署名検証(HMAC-SHA256)                      │
        │  /dl(GET)      ─────────►  ├─ ルーティング（登録/提出/照会/督促/名簿CRUD）│
        ▼                │          ├─ DynamoDB: roster / employees / submissions │
   社員・人事の端末       │          └─ S3: 提出物・様式・zip（署名付き短縮DLへ302）│
                          │                                                     │
   EventBridge(cron) ───────────►  reminder Lambda ──► LINE push（未提出者のみ） │
                          │   SSM Parameter Store(SecureString): LINE 資格情報    │
                          └─────────────────────────────────────────────────────┘
```
時刻は UTC 保存・JST 表示。LINE は署名検証のみで接続でき、**中継・暗号・可信IP は不要**（研修アシスタントの微信客服と異なる点）。

## コンポーネント
| 種別 | リソース | 役割 |
|---|---|---|
| 入口 | **Lambda Function URL** | LINE webhook（POST）と短縮DL（GET /dl）を 1 本で受ける。API Gateway 不要。 |
| 計算 | **webhook Lambda** | 署名検証→イベント解析→業務ルーティング→DynamoDB/S3。512MB（zip生成余裕）。 |
| 計算 | **reminder Lambda** | 定時/手動トリガで未提出者へ LINE push。 |
| データ | **DynamoDB roster** | 社員名簿（主データ・人事管理）。PK=empId。 |
| データ | **DynamoDB employees** | LINE 連携/会話状態。PK=userId（line:xxx）→ empId/status。 |
| データ | **DynamoDB submissions** | 提出記録。PK=userId, SK=`{period}#{type}`、GSI1=`{period}#{type}`→userId。 |
| 保管 | **S3** | 提出物・様式・zip。BlockPublic＋SSE＋バージョン管理。短期署名URL/302で配布。 |
| 機密 | **SSM SecureString** | LINE channel secret / access token（git・環境変数平文に置かない）。 |
| 定時 | **EventBridge Rule** | 月次リマインド（cron、既定 25・28 日 09:00 JST）。 |

## データモデル（要点）
- **roster（主データ）**：`empId, name, department, role(employee|hr), lineUserId?, createdAt`。「誰が提出対象か」を決める。
- **employees（連携）**：`userId, status(awaiting_name|active), empId, name`。LINE 登入＝名簿との紐付け（氏名照合）。
- **submissions**：`userId, sk={period}#{type}, gsi1pk={period}#{type}, gsi1sk=userId, period, type, s3Key, fileName, submittedAt`。
- **未提出者の算出**：roster 全員 −（GSI1 で得た提出者 lineUserId 集合）。未連携者は常に未提出として表示（`※Line未登録`）。

## S3 レイアウト
```
hr/{yyyy}/{mm}/worktimes/作業時間記録簿（氏名）_YYYYMM.xlsx   ← 勤怠
hr/{yyyy}/{mm}/expenses/経費（氏名）_YYYYMM.xlsx              ← 通勤費
hr/template/作業時間記録簿.xlsx / hr/template/経費.xlsx        ← 空白様式
pending/…（種別未判定の一時置き）  exports/…（zip）
```
提出物には `lifecycle=managed` タグを付与し、**60 日で Deep Archive・365 日で削除**（タグ条件のためテンプレ/zip は対象外）。

## ダウンロードの仕組み（短縮リンク）
Lambda の一時認証情報で生成する署名付きURLは 1,000 文字超になり LINE ボタン制限を超えるため、ボタンは自前の短縮エンドポイントを指す：
- 様式：`GET /dl?type=kintai|commute` → 302 で署名付きURLへ。
- 提出物/zip：`GET /dl?key=<s3key>&sig=<HMAC>` → 署名検証後に 302（越権防止）。

## セキュリティ／設計の勘所
- LINE 資格情報は **SSM SecureString**、Lambda が実行時に取得（git/メモリに残さない）。
- S3 は Block Public Access＋SSE＋バージョン管理。提出物DLは **HMAC 署名付き短縮リンク**で越権防止。
- 生成AI（Bedrock）は既定 OFF。利用時も日本リージョン内 inference profile を想定（データ主権）。
- 外部依存ゼロ（標準ライブラリ＋boto3）＝デプロイが軽く、コールドスタートも小さい。
