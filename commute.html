要員通勤コスト
比較ツール

提案時、各要員から客先現場までの通勤時間・乗換回数・片道運賃・通勤定期代を、LINEに現場の駅名を送るだけで待機中の全要員ぶん一括比較。手作業の個別検索をなくします。

資料をダウンロード 仕組みを見る
LINEで一括比較 AWSサーバーレス 住所は保存せず最寄駅のみ

OUR VALUE

提供できる価値

「どの要員を出すと通勤コストが最小か」を、提案のその場で根拠付きで比較できます。

一括比較

現場駅を1つ送るだけで、待機中の全要員の通勤コストを表で比較。個別検索の手間を削減。

明確な選定基準

「定期代が最安」か「所要時間が最短」かを選択。先頭候補を盲目的に採らず根拠を提示。

差し替え可能

データ取得元を抽象化。暫定はWeb、正式版は認可API（駅すぱあと等）へ業務ロジックを変えず移行。

個人情報配慮

社員情報は住所を保存せず「最寄駅」だけ保持。社内システムの花名册と同じ社員IDで連携。

設計のポイント：同じ現場×同じ最寄駅は一度しか問い合わせず、結果は30日キャッシュ。LINEは「即時ackの後、非同期で結果をpush」するため重い検索でもタイムアウトしません。データ取得元は抽象インタフェースの背後に隠し、業務ロジックを変えずに差し替え可能です。

FEATURES

主な機能

1経路あたり4指標を取得し、戦略で1案に絞って比較表にします。

機能内容
4指標の取得所要時間・乗換回数・片道運賃（IC／きっぷ）・通勤定期代（1・3・6ヶ月）
経路選択戦略「定期代が最安（同額なら時間最短）」／「所要時間が最短」を選択
一括比較現場を指定→待機中の全要員を比較。確定済み（現場アサイン中）は自動除外
入力ゆれの吸収駅名は漢字・かな・カタカナ・ローマ字に対応。中国語表記はAIで日本語駅名へ正規化
住所→最寄駅社員が住所しか分からなくても、ジオコーディングで最寄駅を自動判定（保存は駅名のみ）
花名册一括反映CSV/txtを編集して流すだけで社員の増删改を反映（社内システムと同じ社員ID）

※ 経路候補は複数返るため、明示した戦略で1案を選定（先頭固定で採らない）。定期種別は「通勤」を採用。

HOW TO USE

使い方（LINE）

現場の駅名を送るだけ。数秒で待機中の全要員の比較が返ります。

① 入口は3つ

LINE ／ コマンドライン ／ API（IAM署名）から同じ比較を実行。

② 名簿の維持

花名册CSVを編集→1コマンドで反映。住所行は最寄駅へ自動変換。

③ CSV出力

比較結果はCSVにも出力でき、提案資料へそのまま貼付可能。

PROBLEM → SOLUTION

課題と解決アプローチ

要員ごとに通勤費を個別検索していた
現場駅を送るだけで全要員を一括比較。同一駅はキャッシュで再検索しない
どの経路・どの基準で選んだか曖昧
「定期最安／時間最短」を明示し、経由駅まで提示
暫定データ源にロックインしたくない
取得元を抽象化。正式版の認可APIへ業務ロジックを変えず差し替え
社員の住所は持ちたくない（情報管理）
登録時に最寄駅へ変換し、住所は保存しない

TECH STACK

技術構成

レイヤ採用技術役割
入口LINE Messaging API / Lambda Function URL現場駅の受付（署名検証）・API（IAM認証）
アプリAWS Lambda（Python 3.12）一括比較・経路解析・即時ack＋非同期push
データAmazon DynamoDB社員テーブル・通勤キャッシュ（30日TTL）
経路取得抽象データソース（暫定Web／将来 認可API）時間・乗換・運賃・定期代を取得（差し替え可能）
入力正規化Amazon Bedrock（Claude）中国語等の駅名表記を日本語駅名へ正規化（兜底）
位置情報Amazon Location Service住所のジオコーディング→最寄駅判定
IaCAWS CDK（TypeScript）全リソースをコードで定義・デプロイ

※ 取得元は抽象インタフェース（query→共通の結果型）に統一。新データ源はインタフェース実装のみで追加でき、業務層・社員DBは無改修です。
