コンテナWeb 業務システム

REFERENCE ARCHITECTURE

リファレンスアーキテクチャ

Amazon VPC（マルチAZ）／3層サブネット分離（公開・非公開・分離）。

ユーザー
ブラウザ / App
CloudFront
CDN / HTTPS / 防御
ALB
ロードバランサ
ECS Fargate
アプリ(FastAPI)
Aurora Serverless v2
writer（書込） / reader（読取）
設計のポイント：通信は常に HTTPS の CloudFront 経由。ALB は公開しつつ CloudFront からのみ受信を許可（公網から ALB に直結不可）。DB は出口のない分離サブネットに配置し、読み書きを分離して負荷を分散。Aurora は min=0 でアイドル時に自動停止。

COMPONENTS

各コンポーネントの役割

コンポーネント役割採用理由（お客様メリット）
Amazon CloudFrontエッジ配信・HTTPS終端・キャッシュ高速化・HTTPS・DDoS防御を低コストで実現
Application Load BalancerL7ロードバランサ・振り分け可用性向上・集約・ヘルスチェック
Amazon ECS Fargateサーバー管理不要のコンテナ実行基盤運用負荷を抑えつつ自動スケール・無停止更新
Aurora Serverless v2自動伸縮RDB（読み書き分離）負荷に応じ自動拡張、アイドル時は停止し低コスト

※ いずれも AWS マネージドサービスを採用し、構築・運用コストと障害リスクを最小化しています。

PROBLEM → SOLUTION

課題と解決アプローチ

アクセス急増に耐えられない
Fargate と Aurora Serverless v2 の自動スケールで需要に追従
インフラコストが高い・読めない
Serverless v2＋従量課金、アイドル時は自動停止で無駄を削減
セキュリティに不安がある
ネットワーク分離・最小権限IAM・Secrets Manager・オリジン保護
構築／運用が属人化している
AWS CDK でコード化し、誰でも再現・変更影響を事前確認
リリースで停止・失敗が怖い
ローリングデプロイで無停止更新、ヘルスチェックで自動復旧

NON-FUNCTIONAL

非機能要件への作り込み

「動く」だけでなく、本番運用に求められる品質を標準で作り込みます。

セキュリティ
  • 3層ネットワーク分離
  • ALBはCloudFront経由のみ許可
  • 認証情報はSecrets Manager
  • IAMは最小権限
コスト最適化
  • Serverless v2 Min0 で自動停止
  • 固定費(NAT/ALB)を可視化
  • タグでコスト按分・分析
  • 従量課金で無駄を排除
可用性
  • マルチAZで単一障害点回避
  • ALBヘルスチェックで異常隔離
  • リーダー昇格で自動切替
  • 無停止のローリング更新
運用性
  • AWS CDKでコード化
  • cdk diffで変更影響を事前確認
  • CloudWatchへログ集約
  • 環境の再現・複製が容易

IMPLEMENTATION

実装例：読み書き分離デモ

CDK（TypeScript）単一スタックで全構成をデプロイ／破棄可能。コンテナ内で FastAPI の商品アプリを動かし、Aurora の読み書き分離を可視化します。

  • GET / 商品一覧 → reader 端点（読）
  • POST /products 追加 → writer 端点（書）
  • どのノードで読んだか表示（複製遅延も体感）
  • ALB 直結は安全グループで拒否（入口は CloudFront のみ）
